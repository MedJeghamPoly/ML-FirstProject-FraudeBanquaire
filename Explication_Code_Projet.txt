================================================================================
  EXPLICATION DU CODE — PROJET DÉTECTION DE FRAUDE BANCAIRE (MACHINE LEARNING)
================================================================================

Document de synthèse : objectif pédagogique, aligné sur le dépôt (notebooks,
package src/, configuration). Les métriques chiffrées exactes dépendent de
l'exécution locale (fichier outputs/model_results.csv).


--------------------------------------------------------------------------------
1. PRÉSENTATION GÉNÉRALE DU PROJET
--------------------------------------------------------------------------------

1.1 Objectif
------------
  Classifier automatiquement des transactions de carte bancaires en deux classes :
    • Class = 0 : transaction légitime
    • Class = 1 : transaction frauduleuse

  La sortie utile en production est souvent la probabilité d'être une fraude,
  puis un seuil métier appliqué sur cette probabilité (pas seulement 0,5).


1.2 Contexte et données
------------------------
  • Jeu : "Credit Card Fraud" (ULB), souvent distribué via Kaggle.
  • Fichier attendu : data/creditcard.csv
  • Environ 285 000 transactions, variables numériques :
      - Time, Amount
      - V1 à V28 : composantes issues d'une ACP (données originales anonymisées)
      - Class : cible binaire
  • Le déséquilibre est fort : peu de fraudes par rapport aux transactions
    normales. L'"accuracy" seule est trompeuse si le modèle prédit presque
    toujours "légitime".


1.3 Problème traité (formulation ML)
------------------------------------
  • Apprentissage supervisé : classification binaire.
  • Difficultés principales :
      - déséquilibre des classes ;
      - besoin de métriques adaptées (F1, précision/rappel sur la fraude,
        ROC-AUC, courbe précision-rappel) ;
      - risque de "fuite" d'information si on mélange indûment passé et futur
        (d'où le tri temporel et des features causales).

2. ARCHITECTURE LOGICIELLE (APERÇU)
--------------------------------------------------------------------------------

  • notebooks/01_EDA.ipynb        : exploration (distributions, corrélations)
  • notebooks/02_Modeling.ipynb   : pipeline complet (features, modèles, seuils)
  • src/                          : code réutilisable
      - paths.py      : chemins, RANDOM_STATE, tailles de test, CV
      - config.py     : validation temporelle, coûts FN/FP, options L1, SHAP, etc.
      - feature_engineering.py : log1p(Amount), ratios sur moyenne mobile causale
      - selection.py  : sélection de variables par coefficients L1
      - splits.py     : découpage temporel ordonné
      - ml_utils.py   : pipelines SMOTE + scaler + classifieur, métriques
      - drift.py, thresholds.py, calibration_plots.py, shap_utils.py : analyses
  • outputs/ : figures, model_results.csv, best_model.pkl

  Chaîne type dans le notebook :
    chargement → features → split train/test → (sélection L1) → pipelines avec
    standardisation + SMOTE sur le train → entraînement de plusieurs modèles →
    comparaison sur le jeu de test → analyses (PR/coût, calibration, dérive, SHAP).


--------------------------------------------------------------------------------
3. MODÈLES UTILISÉS ET CHOIX DU MODÈLE "RETENU"
--------------------------------------------------------------------------------

3.1 Modèles comparés (dans le code)
-----------------------------------
  Le projet n'impose pas un seul algorithme : plusieurs estimateurs sont entraînés
  et comparés sur le jeu de test (et la validation croisée sur le train) :

    • Régression logistique (lbfgs, fort régularisée implicitement par la pipeline)
    • Arbre de décision (profondeur plafonnée)
    • Forêt aléatoire (Random Forest), avec recherche d'hyperparamètres (GridSearch)
    • Gradient boosting : XGBoost si la bibliothèque est installée, sinon
      HistGradientBoostingClassifier (scikit-learn)
    • LightGBM (optionnel, si le package est disponible)

3.2 Pourquoi ce panel de modèles ?
----------------------------------
  • Baselines interprétables : régression logistique et petit arbre servent de
    références simples.
  • Modèles d'ensemble (forêt, boosting) : adaptés aux données tabulaires, captent
    des interactions non linéaires entre V* et Amount, souvent meilleurs F1 / AUC
    sur ce type de benchmark.
  • Comparer plusieurs familles permet de documenter un vrai choix de modèle
    (pas un seul algorithme "imposé").

3.3 Comment le "meilleur" modèle est choisi dans le notebook
------------------------------------------------------------
  Après calcul des métriques sur le test, les modèles sont triés principalement
  selon le F1 (puis ROC-AUC en secours). Le premier de ce classement devient
  "best_name" : c'est lui qui sert aux courbes ROC/PR, calibration, matrice de
  confusion, export best_model.pkl et SHAP (quand applicable).

  En pratique, sur ce jeu, la forêt aléatoire bien réglée est souvent parmi les
  meilleurs compromis précision/rappel ; ce n'est pas une règle absolue : il faut
  lire outputs/model_results.csv après chaque exécution.


--------------------------------------------------------------------------------
4. ANALYSE DES PERFORMANCES (LECTURE GÉNÉRALE)
--------------------------------------------------------------------------------

4.1 Points forts (à mettre en regard des sorties réelles)
-----------------------------------------------------------
  • Random Forest / boosting peuvent atteindre un bon équilibre entre détecter
    les fraudes (rappel) et limiter les alertes inutiles (précision), mieux qu'un
    simple seuil sur la majorité.
  • La validation temporelle (option activée) rend l'évaluation plus réaliste
    qu'un tirage aléatoire stratifié : le modèle est testé sur des instants
    "futurs" par rapport au train.
  • Le pipeline SMOTE uniquement sur le train limite le sur-apprentissage de la
    sur-représentation artificielle de la classe minoritaire sur le test.
  • Métriques multiples (F1, ROC-AUC, PR, coût métier) : cohérentes avec un
    cahier des charges "fraude".

4.2 Limites et inconvénients
----------------------------
  • Données anciennes (contexte 2013) : la distribution des transactions réelles
    actuelles peut différer (dérive). Les analyses KS/PSI dans le projet alertent
    sur des écarts train/test mais ne "corrigent" pas le modèle.
  • Variables V* sont des composantes d'ACP : interprétation métier directe limitée ;
    SHAP aide à hiérarchiser l'importance des variables du modèle, pas le sens
    métier brut.
  • Optimiser le F1 sur le test ne garantit pas la généralisation : le jeu test
    est une seule coupe temporelle ; en production, il faudrait validation sur
    d'autres périodes ou monitoring continu.
  • SMOTE crée des exemples synthétiques : utile pour entraîner, mais les
    frontières peuvent être optimistes si mal calibrées.
  • Coût FN/FP dans config (COST_FN, COST_FP) est indicatif : les vrais coûts
    métiers doivent être fixés avec les équipes risques.


--------------------------------------------------------------------------------
5. PARAMÈTRES ET HYPERPARAMÈTRES (RÔLE, IMPACT, VALEURS ACTUELLES)
--------------------------------------------------------------------------------

Les valeurs ci-dessous correspondent au code (src/paths.py, src/config.py,
scripts/emit_02_notebook.py / notebook 02_Modeling).

5.1 Découpage des données
-------------------------
  TEST_SIZE = 0,2 (paths.py)
    • Rôle : fraction du jeu réservée au test (en temporal split : derniers 20 %
      des lignes après tri par Time).
    • Impact : augmenter le test réduit le train (variance plus forte sur les
      métriques) ; diminuer le test donne plus de données pour apprendre mais
      moins pour évaluer.
    • Justification : 80/20 est un standard raisonnable avec ~285k lignes.

  USE_TEMPORAL_SPLIT = True (config.py)
    • Rôle : train = premiers instants, test = derniers (via Time).
    • Impact : souvent F1/précision différents d'un split stratifié aléatoire ;
      plus réaliste pour une fraude liée au temps.
    • Justification : aligné avec l'objectif "scoring" dans le temps.

  RANDOM_STATE = 42 (paths.py)
    • Rôle : fixe le hasard (SMOTE, sous-échantillons, CV, etc.).
    • Impact : reproductibilité ; changer la graine change légèrement les résultats.
    • Justification : convention pour des expériences reproductibles.


5.2 Validation croisée et sous-échantillonnage pour la grille RF
-----------------------------------------------------------------
  CV_FOLDS = 2
    • Rôle : nombre de plis pour la GridSearch de la forêt aléatoire.
    • Impact : 2 plis = rapide mais estimateur de performance plus variable ;
      5 ou 10 serait plus stable mais plus lent sur gros volumes.
    • Justification : compromis temps CPU / projet pédagogique.

  GRIDSEARCH_TRAIN_SAMPLES = 25 000
    • Rôle : taille du sous-échantillon stratifié utilisé uniquement pour la
      recherche des hyperparamètres RF.
    • Impact : accélère la grille ; si trop bas, la grille peut sous-estimer la
      performance réelle sur tout le train.
    • Justification : réduire le temps de calcul tout en gardant des fraudes
      représentées (stratification).

  CV_SCORING_FOLDS = 3 et CV_SCORING_MAX_SAMPLES = 35 000
    • Rôle : CV pour afficher des scores F1/ROC moyens sur le train pour chaque
      modèle (hors grille RF).
    • Impact : même logique — équilibre stabilité / temps.


5.3 Feature engineering (feature_engineering.py)
-------------------------------------------------
  Fenêtre rolling pour Amount : w = 5000
    • Rôle : moyenne mobile sur les transactions passées (shift(1)) pour éviter
      toute fuite du futur ; ratio Amount / (moyenne passée).
    • Impact : fenêtre grande = lissage fort ; petite = plus réactif mais bruité.
    • Justification : ordre de grandeur lié à la granularité temporelle implicite
      du jeu (paramètre fixé dans le code comme compromis).

  log1p(Amount)
    • Rôle : réduit l'asymétrie des montants ; stabilise l'échelle pour les modèles.
    • Impact : souvent meilleure convergence pour LR et arbres.


5.4 Sélection de variables L1 (selection.py + config)
-------------------------------------------------------
  ENABLE_FEATURE_SELECTION = True
  L1_MAX_FEATURES = 25
  L1_SAMPLE_ROWS = 40 000
  Logistique L1 : solver liblinear, l1_ratio=1, C=0,1, class_weight=balanced
    • Rôle : entraîner une logistic sparse sur un sous-échantillon, garder les K
      plus grandes valeurs absolues de coefficients (classe positive).
    • Impact :
        - C plus petit = plus de régularisation = souvent moins de variables non
          nulles (plus sparse) ;
        - C plus grand = coefficients moins pénalisés ;
        - max_features limite la dimension finale ;
        - balanced pondère les classes pour la sélection malgré le déséquilibre.
    • Justification : réduit le bruit et la dimension ; C=0,1 et K=25 sont des
      choix pragmatiques pour un projet de taille fixe (ajustables si sous/sur-fit).


5.5 Pipeline SMOTE + classifieur (ml_utils.py)
----------------------------------------------
  SMOTE(random_state=RANDOM_STATE) après standardisation, sur le train seulement.
    • Rôle : sur-échantillonner la classe minoritaire dans l'espace transformé.
    • Impact : si mal utilisé sur le test, fuite d'information ; ici le pipeline
      est respecté (test non sur-échantillonné).


5.6 Forêt aléatoire — grille recherchée
---------------------------------------
  param_grid_rf :
    clf__n_estimators ∈ {100, 200}
    clf__max_depth ∈ {25, None}

  n_estimators
    • Rôle : nombre d'arbres ; moyenne des votes / probabilités.
    • Impact : plus d'arbres = souvent meilleure variance généralisée mais temps
      CPU et mémoire plus élevés ; au-delà d'un certain point, gain marginal.

  max_depth
    • Rôle : profondeur maximale de chaque arbre ; None = jusqu'à séparation pure
      (sous contraintes min_samples_* par défaut sklearn).
    • Impact : profondeur faible = moins de sur-apprentissage mais plus de biais ;
      profondeur élevée = modèle plus expressif mais risque de sur-apprentissage.

  scoring="f1" dans GridSearchCV
    • Rôle : choisit les hyperparamètres selon le F1 (adapté au déséquilibre).
    • Justification : mieux aligné qu'une accuracy pour la classe minoritaire.


5.7 Autres modèles (valeurs fixes dans le notebook)
----------------------------------------------------
  Régression logistique : max_iter=5000, solver=lbfgs
    • max_iter : nombre max d'itérations d'optimisation ; augmenter si convergence
      lente.
    • lbfgs : bon défaut pour problèmes lisses sans L1 sur le modèle final pipeline.

  Arbre de décision : max_depth=10
    • Limite la profondeur pour limiter le sur-apprentissage sur données bruitées.

  XGBClassifier (si installé) : n_estimators=150, max_depth=6, learning_rate=0,08,
    subsample=0,9, colsample_bytree=0,9
    • learning_rate : pas du boosting ; plus petit = souvent besoin de plus
      d'arbres mais généralisation parfois meilleure.
    • subsample / colsample_bytree : sous-échantillonnage lignes/colonnes à chaque
      arbre — réduit le sur-apprentissage.

  HistGradientBoostingClassifier (fallback) : max_iter=150, max_depth=7,
    learning_rate=0,08

  LightGBM (optionnel) : n_estimators=200, max_depth=-1 (pas de limite par défaut
    LGBM), learning_rate=0,05, subsample=0,9
    • learning_rate plus bas que XGB dans ce code : apprentissage plus prudent,
      souvent avec plus d'arbres pour compenser.


5.8 Coûts métier et seuil (config.py + thresholds)
-------------------------------------------------
  COST_FN = 10, COST_FP = 1
    • Rôle : pondérer une fraude manquée (FN) dix fois plus qu'une fausse alerte
      (FP) dans la recherche de seuil sur les probabilités.
    • Impact : pousse à un seuil plus bas (plus de détection de fraude) si le
      coût FN est élevé — au prix de plus de FP.
    • Justification : exemple typique "une fraude coûte plus cher qu'une alerte à
      vérifier" ; à recalibrer avec le métier réel.


5.9 SHAP (config)
-----------------
  ENABLE_SHAP = True, SHAP_MAX_BACKGROUND = 2000, SHAP_MAX_EXPLAIN = 500
    • Rôle : expliquer un sous-échantillon de prédictions (temps raisonnable).
    • Impact : augmenter explique mieux la variabilité mais allonge le calcul.


--------------------------------------------------------------------------------
6. SYNTHÈSE : POURQUOI LES VALEURS ACTUELLES SONT COHÉRENTES
--------------------------------------------------------------------------------

  • Déséquilibre + métier : F1 pour la grille RF, pondération balanced en L1,
    SMOTE sur le train, coûts FN/FP pour le seuil.
  • Temps de calcul : sous-échantillons pour GridSearch et CV, CV_FOLDS=2 pour la
    grille RF — adaptés à un environnement étudiant / machine standard.
  • Risque de fuite : split temporel, rolling causal sur Amount, pas de SMOTE sur
    le test.
  • Le "meilleur" modèle n'est pas figé dans le code : il est le résultat du
    classement sur le test ; la documentation des hyperparamètres ci-dessus permet
    d'expliquer ce qui a été exploré et ce qui reste fixe.

Pour toute évolution : ajuster d'abord COST_FN/COST_FP avec le métier, puis
max_depth / n_estimators / learning_rate si sous-apprentissage ou sur-apprentissage
visible sur courbes PR et calibration, en gardant une validation temporelle ou
des plis temporels si les données le permettent.
